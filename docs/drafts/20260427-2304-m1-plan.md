## 1. Goal & Success Criteria

**Goal.** Ship a TypeScript + Vite + React demo at `/home/pavel/work/safe/scroll-demo` of an index-space-scrolled chat window backed by an in-process mock backend, supporting millions of messages, ID-anchored jumps, region-based memory eviction, a custom approximate scrollbar, search, day-grouped sticky headers, live-tail messages, and a jump-to-latest pill ‚Äî with zero visible flicker during scroll, prepend, measurement, eviction, or merge.

**Acceptance criteria (M1).** Each is independently verifiable.

1. **AC-1 No flicker.** Manual scroll up/down and prepend/append cycles for 60s show no visual jump > 1px on visible rows. Verified by recording a short screen capture and stepping through frames; no row's `top` value changes between consecutive frames except as a result of user input or live-arrival animation.
2. **AC-2 Paginated load.** Backend `getRange` calls are issued only for index ranges not already loaded, with simulated 100‚Äì300 ms latency. Network panel (instrumented via a `MockBackend` call log surfaced in a debug badge) shows ranges fetched on demand and never duplicated for the same span concurrently.
3. **AC-3 Variable heights.** Generated content includes 1‚Äì20-line messages and code blocks. Visual inspection confirms heights vary; `ChatStore.heightsMap` shows >5 distinct measured heights within any 100 visible rows.
4. **AC-4 ID anchors.** `Jump to ID` UI (debug field) takes a stable `msg-NNNNNN` ID and lands the message centered (or first-fully-visible) in viewport, even when the target index lives outside any loaded region. Round-trip ‚â§ 500 ms p95.
5. **AC-5 Bounded memory.** After a sequence of 20 random jumps across N=5,000,000, `ChatStore.regionCount` stays ‚â§ 4 and `ChatStore.totalLoadedMessages` stays ‚â§ 4 √ó (overscan window) once the eviction debounce fires. Surfaced via a debug badge.
6. **AC-6 Scalable scrollbar.** With N=5,000,000 the custom scrollbar thumb is draggable end-to-end; thumb position equals `topIndex / N` within ¬±1 px; click-on-track pages by viewport height. Thumb minimum size honored (e.g., 24 px).
7. **AC-7 Search.** Debounced search (‚â• 250 ms) hits `MockBackend.search`; clicking a result jumps to that ID (re-uses AC-4 path).
8. **AC-8 Sticky day headers.** While scrolling, the date for the topmost visible message is shown stuck to the top of the viewport; transition between dates does not flash empty/duplicate headers.
9. **AC-9 Live tail + jump pill.** New messages arrive at tail every few seconds (subscription). When the user is not anchored at the bottom, an "N new" pill appears; clicking it scrolls to the absolute tail and resumes auto-anchor.
10. **AC-10 Strict TypeScript.** `pnpm tsc --noEmit` passes with `strict: true` and no `any`/`unknown` casts that hide real errors.
11. **AC-11 Initialization.** App boots in < 500 ms on dev server with N=5,000,000 (no upfront fetch of all data; only `getTotalCount` + initial `getLatest`).

**Data flows.**

- **Scroll-down with fetch.** Wheel/touch event ‚Üí `ChatStore.applyDelta(deltaPx)` ‚Üí walks `topIndex/pixelOffset` forward consuming heights from `heightsMap` (fallback estimate). On each tick, `ChatStore` computes the visible range `[topIndex, topIndex + visibleCount + overscan]`. If any sub-range falls outside loaded regions, `ChatStore.ensureRange(start,end)` enqueues a fetch (coalesced; in-flight ranges tracked). On resolve, region is inserted/merged; React subscribers re-render. No layout shift because new rows go below the viewport edge or into the lower overscan band.
- **Jump-to-id.** UI calls `ChatStore.jumpToId(id)` ‚Üí `MockBackend.getById(id)` ‚Üí returns `{index, before, after}` ‚Üí store atomically: drops nothing yet (eviction is debounced), inserts the new region around `index`, sets `topIndex = index`, `pixelOffset = 0`. Re-render. Debounced evictor then drops far regions.
- **Eviction.** A debounced (e.g., 750 ms after `topIndex` settles) pass walks `regions[]`; any region whose `[start,end]` is wholly more than `K=500` rows from `topIndex` (and which is not the tail-anchored region while the user is anchored to bottom) is dropped. In-flight fetches whose target is now far are aborted (AbortController). Heights for dropped indices are cleared from `heightsMap`.
- **Live message arrival.** `MockBackend.subscribeNew` fires `{message, newIndex}`. Store appends to the tail region if loaded; increments `N`. If user is bottom-anchored, advance `topIndex` to keep the new tail visible (or rely on the natural following). Otherwise increment a `unseenCount` and surface via `JumpToLatest`.
- **Scrollbar drag-jump.** `CustomScrollbar` translates pointer Y ‚Üí fractional position ‚Üí target index `Math.round(frac * (N-1))`. Calls `ChatStore.setTopIndex(target, 0)`. Region around target is fetched via `ensureRange`. Re-render shows skeleton rows (estimate-height placeholders) until fetched (‚â§ 300 ms simulated). No flicker because skeletons occupy estimated space and resolve in place.

## 4. Cross-Cutting Architectural Notes

- **Library versions (latest stable as of 2026-04-27).** React 19.x, React-DOM 19.x, TypeScript 5.9.x (strict + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes`), Vite 6.x with `@vitejs/plugin-react` 4.x, Vitest 2.x, jsdom 25.x, @testing-library/react 16.x. (Pin in `package.json`; if any has shifted, take the current latest stable ‚Äî no betas.) No UI framework, no virtualization library.
- **State management.** Plain `ChatStore` class implementing a minimal observable (`subscribe(listener)`/`getSnapshot()`). React layer integrates via `useSyncExternalStore`. Justification: the scroll engine demands fine-grained, synchronous-on-input updates that don't fit React's reducer flow well; a class lets us mutate `topIndex/pixelOffset` per wheel event with one notify at end. Avoids context overhead and unnecessary re-renders.
- **Testing.** Vitest in jsdom. Pure-logic tests (`regions`, `ChatStore` fetch coordination, eviction, anchor math) covered comprehensively. ResizeObserver is unavailable in jsdom ‚Äî provide a tiny polyfill in `src/test/setup.ts` that no-ops; component tests that need real measurements are out of scope for unit tests and verified manually. Minimum: ‚â• 90% branch coverage on `regions.ts` and `ChatStore`.
- **CSS.** Plain CSS files imported by component (one `.css` per component). No CSS modules ‚Äî overkill for this small surface. No CSS-in-JS ‚Äî avoid runtime cost. Reset minimal; class names prefixed by component (e.g., `.chat-viewport__row`).
- **"Scroll-settled" detection.** Wheel/key/drag events update `topIndex/pixelOffset` synchronously. After every input, schedule a 150 ms `setTimeout` that, if not cancelled by another input, marks `settled=true` and triggers (a) `ensureRange` for current visible+overscan window, and (b) the eviction debouncer (750 ms after settled). rAF is used only for the layout commit pass (batch height-map updates).
- **Eviction √ó in-flight fetches.** Each fetch carries an `AbortController`; on eviction or large jump, controllers for ranges no longer needed are aborted. Resolved fetches whose target range is no longer "needed" (region was evicted before resolution) are dropped silently ‚Äî `fetchCoordinator` checks at resolve time whether the requesting topIndex window still overlaps.
- **Concurrency / coalescing.** `ensureRange(start,end)` walks `regions` + `inflight` to compute the missing sub-ranges; chunks into windows of ‚â§ 100; for each window, registers an `AbortController` keyed by `rangeKey = startIndex` and skips if a key already exists. Same `rangeKey` is never fetched twice.
- **Coordinate-system invariants (reviewers, check these on every PR).**
  - I-1 `pixelOffset ‚àà [0, heightOf(topIndex))` always after any state-mutating action.
  - I-2 Layout walks downward from `topIndex`. Rows above viewport edge are not laid out (only retained for measurement if necessary; we don't need that ‚Äî measurement happens when the row is in the overscan band).
  - I-3 `regions[]` are sorted by `startIndex`, disjoint, non-empty.
  - I-4 `heightsMap` only contains entries for indices that are currently in some region OR currently in viewport+overscan.
  - I-5 `tailAnchored` derived, never stored.
  - I-6 Scrollbar thumb top depends solely on `topIndex` and `N`.
- **Wheel-event hijack.** We hijack: `wheel` listener with `passive: false` on the viewport, `preventDefault()` always. Trackpad inertia comes through naturally as a stream of small `deltaY`s; we don't need a momentum simulator. We do clamp per-event `deltaY` to viewport height to avoid overshoot from rogue events.
- **Touch.** `touchstart/move/end` translate finger movement to `pixelOffset` delta. No momentum simulation in M1 (out-of-scope mobile-specific gesture).
- **Determinism.** All randomness (content generation, latency) is seeded; the seed is configurable via URL param `?seed=N` for reproducible bug reports.

## 5. Risks & Assumptions

- **R-1 Async height changes.** ResizeObserver fires once after element appears. If a row's content loads asynchronously (we don't have images in M1, but code-blocks could re-layout via fonts), a second fire could shift content. Mitigation: rows above viewport never affect layout (we layout downward), so async height changes only affect rows at or below the topRow ‚Äî in which case a downward reflow is fine. For the topRow itself, we keep its top edge fixed and let later rows reflow.
- **R-2 Trackpad/Firefox wheel deltas.** Firefox uses `deltaMode = LINE/PAGE`. We multiply by configured line-height (‚âà estimate height) for LINE and viewport height for PAGE. Documented in code; covered by manual cross-browser test.
- **R-3 Wheel passive default.** In React 18+, listener attached via JSX `onWheel` is passive by default; `preventDefault` won't work. We use `useEffect` + `addEventListener('wheel', handler, { passive: false })`. Critical.
- **R-4 Sticky header z-index/clipping.** Header lives inside `overflow:hidden` viewport; it's `position:absolute; top:0`. Risk: it gets clipped by row stacking context. Mitigation: header is a sibling of the rows-container, not a child of any row; explicit `z-index: 10`.
- **R-5 Search over 1M+ messages.** Linear scan against the generator is O(N) per query. Mitigation: bound search to 5,000-result budget and short-circuit; debounce 300 ms; show "showing first 50 results" footer. Acceptable for a demo.
- **R-6 Regenerating content for `getById`.** Since content is deterministic by index, `getById('msg-001234')` parses the index from the ID and regenerates locally ‚Äî no full scan. Document this as part of the PRNG contract.
- **R-7 ResizeObserver in jsdom.** Polyfill in test setup; integration coverage of layout deferred to manual.
- **R-8 Browser scrollHeight cap (the reason we exist).** Our viewport DIV never grows beyond viewport-height. We never set `height: N * estHeight`. This is the central architectural commitment.
- **R-9 Live tail vs. eviction race.** If the user scrolls away from tail, the tail region becomes evictable. If a new message arrives, we must extend or reinsert the tail region. Handled by a `lastEmittedTail` cache: store always retains the latest tail message metadata even when its region was evicted, so re-insert is correct.
- **A-1** N is fixed at app boot (plus live-tail increments). No mid-run re-numbering.
- **A-2** All messages have a single timeline; no threads/channels.
- **A-3** Avg row height ~ 60 px is "good enough" for thumb sizing; overshoot < 1 viewport per 1000 rows is acceptable.
