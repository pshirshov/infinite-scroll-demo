# scroll-demo M1 — Session Log (2026-04-27)

## Original user request

> Let's setup a nix environment and create a typescript+vite+react demo of an
> endlessly scrollable chat window. Key requirements:
> 1) no flickering or blinking
> 2) paginated loading
> 3) messages of different length (thus different message height)
> 4) we need anchors and ability to quickly jump to a particular message
>    even if it's not loaded
> 5) efficient memory management — if we jump to a page far away, we shouldn't
>    consume all the ram
> 6) consistent scrollbar behavior
>
> Research prior art, ask your questions if needed then work in /review-loop

After clarifying questions, the user pinned: known total count up-front, mock
data only, jump by stable ID, standard chat UX (newest at bottom, live tail),
hand-rolled (no react-virtuoso / @tanstack/react-virtual / react-window),
efficient eviction with optional ROM-style scrollbar. Critically, the user
rejected my initial demo cap and clarified: "We want to support millions! Hint:
the scrollbar should not be exact, it may be a rough approximation of the
timeline." This redirected the architecture from "tall container + native
scrollbar" to **index-space scroll** with a **custom scrollbar**, which became
the central design commitment.

## Architectural commitments worth highlighting for future readers

These are the design decisions that ended up load-bearing across multiple PRs
and are most likely to surprise someone reading the code cold:

1. **Index-space scroll, not pixel-space.** The viewport never creates a
   `N × estHeight` div. Scroll state is `(topIndex, pixelOffset)`. This is
   the foundational decision that makes N = 5 M tractable. Browsers cap
   scroll height around 17–33 M px; we sidestep that entirely.

2. **Layout flows DOWNWARD from `topIndex`.** Above-rows are derived backward
   from `topIndex`'s top. Height changes above the fold never affect visible
   content. This is invariant I-2 and is the source of the no-flicker
   guarantee.

3. **`borderBoxSize`, NEVER `contentRect.height`** for row measurements.
   PR-04-D01 documented the consequence of getting this wrong.

4. **`ChatStore.getSnapshot()` returns a `===`-stable reference between
   mutations.** Pure `setHeight` calls don't invalidate the regions array
   reference. This matters because React effects with `[snap.regions]` deps
   would otherwise re-fire on every measurement — causing the initial-anchor
   effect to retrigger or eviction to thrash.

5. **The `disposed` flag in `FetchCoordinator`** guards both `.then` (to
   prevent stale-region insertion) and `.catch` (to suppress onError floods
   during teardown). Plus `.finally` for cleanup. Same pattern repeated in
   `SearchBar`. Any future async resource should follow it.

6. **`applyScrollDelta` is the SINGLE mutation path** for scroll state
   (invariant I-4). Wheel, keyboard, scrollbar drag, jump-to-id, jump-to-
   latest, initial anchor — all funnel through it. Don't add side channels.

7. **Custom scrollbar is a rough timeline approximation** by user direction.
   Thumb at `topIndex / (N-1)`; size clamped to 24 px min. No accumulated
   pixel drift even at N = 5 M.

## Closing note on review discipline

Two operational lessons emerged from the session:

- Round-2 reviews are necessary for major fixes (PR-04, PR-05) to confirm
  the patch worked AND didn't regress. Reviewer budget is real; the
  PR-05 round-2 review hit its limit and the orchestrator self-verified.
  In hindsight, more frugal reviews on mechanical PRs (PR-06 onwards)
  preserved budget for the consequential ones.

- A clean GREEN verdict means **merge**, not "investigate further nits."
  The skill warned about this; I drifted on PR-03 by launching a
  follow-up fix pass for non-blocking minor nits. From PR-06 onwards I
  closed cleanly on GREEN verdicts and recorded notes in defects.md.
