## Defects fixed

- **PR-14-D01 (major, user-reported)** — DOM-order vs visual-order mismatch
  in `ChatViewport.tsx` layout pass made browser drag-selection traverse
  unrelated overscan rows. Fixed via explicit sort. Codified as I-7.
- **PR-15-D01 (major, user-reported)** — `useState(() => new
  ChatStore(...))` initializer + `useEffect(() => () => store.dispose(),
  [store])` cleanup interacted incorrectly with React 18 `<StrictMode>`.
  The cleanup disposed the store; the remount reused the same disposed
  store; every coordinator-resolved prefetch was silenced by the
  `disposed` race-guard from PR-05's resolved-after-aborted fix. Fixed
  by moving resource creation into an effect.

## Why M1 unit tests didn't catch either bug

Both bugs only manifest in the React lifecycle layer:

- **PR-14-D01** is a DOM-order property of the rendered tree. Unit tests
  exercise pure logic (regions, scroll math, fetch coordinator) and never
  walk a real DOM.
- **PR-15-D01** is an interaction between React's StrictMode dev-mode
  double-mount and the dispose cleanup. Unit tests instantiate
  `ChatStore` and `FetchCoordinator` directly without React; the
  StrictMode pattern is invisible.

Both surfaced **only** under user manual testing (the user's report) AND
under Playwright e2e (which exercises the real React tree, including
StrictMode in dev). M2 closes this integration-coverage gap.

## Architectural lessons worth highlighting

1. **DOM order MUST match visual order** when `transform: translateY` is
   used with absolute positioning AND text selection is desired.
   Codified as invariant I-7 in `ChatViewport.tsx`.

2. **Resources whose lifetime tracks the React tree MUST be created in
   an effect, NEVER via `useState(() => new Resource())` when paired
   with a dispose cleanup.** Otherwise React 18 StrictMode's
   intentional double-mount pattern reuses the disposed instance.
   Pattern documented inline in `App.tsx`. Worth codifying in
   CONTRIBUTING if M3+ adds more such resources.

3. **PR-05's round-1 review explicitly flagged the StrictMode
   interaction** but mis-analysed the consequence ("useState
   initializer only runs once per mount instance, so the second mount
   creates a fresh store" — incorrect). Stronger reviewer modelling of
   React internals would have caught it. Lesson for future review
   prompts: when reviewing dispose lifecycles, name the StrictMode
   double-mount case explicitly and ask the reviewer to trace the
   `useState` semantics step-by-step.

4. **Test-first works.** Both PR-14 and PR-15 wrote a failing
   Playwright test BEFORE the fix. PR-15 in particular benefited
   enormously: the diagnostic logs against the failing test pointed
   directly at the StrictMode pathology. Without the failing repro,
   "empty bodies after drag" is hand-wavy; with it, the timer logs
   and `disposed=true` evidence are unambiguous.

5. **Playwright via nix is sandbox-friendly.**
   `nixpkgs#playwright-driver.browsers` + matching `@playwright/test`
   version → no `npx playwright install` step, no host-cache pollution,
   reproducible across machines that share the flake.
